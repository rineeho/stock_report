# Feature Specification: Daily Stock Report Agent System

**Feature Branch**: `001-daily-report-agent`
**Created**: 2026-04-10
**Status**: Draft
**Input**: User description: "여러 금융/증권사 사이트에서 매일 발행되는 주식 리포트를 자동으로 수집하고, 당일 리포트만 선별하여 정리하는 agent 기반 시스템"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 당일 리포트 일괄 수집 및 열람 (Priority: P1)

사용자는 특정 날짜를 지정하여 시스템을 실행하면, 승인된 사이트 목록에서 해당 날짜에 발행된 주식 리포트를 자동으로 수집하고, 날짜가 검증된 리포트만 정리된 목록으로 확인할 수 있다. 사용자는 더 이상 여러 사이트를 직접 방문하여 리포트를 하나씩 확인할 필요가 없다.

**Why this priority**: 시스템의 핵심 가치. 수집과 날짜 검증이 없으면 이후 모든 기능(요약, 분류 등)이 동작할 수 없다.

**Independent Test**: 승인된 사이트 2~3개에 대해 특정 날짜로 실행한 뒤, 결과 목록에 해당 날짜 리포트만 포함되어 있는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 승인된 사이트 목록이 설정되어 있고 오늘 날짜가 2026-04-10(KST)일 때, **When** 사용자가 target_date=2026-04-10으로 수집을 실행하면, **Then** 각 사이트에서 2026-04-10에 발행된 리포트만 수집되어 목록으로 표시된다.
2. **Given** 특정 사이트에서 날짜를 추출할 수 없는 리포트가 있을 때, **When** 수집이 완료되면, **Then** 해당 리포트는 최종 결과에서 제외되고 "검증 필요(UNVERIFIED)" 상태로 별도 표시된다.
3. **Given** 리포트의 발행일이 target_date와 다를 때, **When** 날짜 검증을 수행하면, **Then** 해당 리포트는 결과에서 제외된다.
4. **Given** 수집된 각 리포트에 대해, **When** 결과 목록을 확인하면, **Then** 각 항목에 제목, 증권사, 애널리스트, 발행일, 원본 URL이 포함되어 있다.

---

### User Story 2 - 리포트 핵심 요약 확인 (Priority: P2)

사용자는 수집된 각 리포트에 대해 핵심 요약을 확인할 수 있다. 요약에는 주요 내용, 투자 의견, 목표주가 등 핵심 포인트가 포함되며, 원문에 없는 정보는 생성되지 않는다.

**Why this priority**: 수집된 리포트를 하나씩 읽지 않고 빠르게 핵심을 파악할 수 있는 가치를 제공한다. 수집(P1)이 완료되어야 요약이 가능하다.

**Independent Test**: 수집된 리포트 5건에 대해 요약을 생성한 뒤, 각 요약이 원문 정보만 포함하는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 날짜 검증을 통과한 리포트가 존재할 때, **When** 요약 생성이 완료되면, **Then** 각 리포트에 주요 내용, 투자 의견, 핵심 포인트를 포함한 요약이 생성된다.
2. **Given** 리포트에 목표주가 정보가 없을 때, **When** 요약을 생성하면, **Then** 목표주가 필드는 "정보 없음"으로 표시되며, 임의 수치가 생성되지 않는다.
3. **Given** 요약 결과물에 대해, **When** 사용자가 확인하면, **Then** 원문에서 직접 추출한 정보(extracted)와 생성된 요약(generated)이 구분되어 표시된다.

---

### User Story 3 - 중복 리포트 통합 (Priority: P3)

사용자는 여러 사이트에서 동일한 리포트가 중복 수집된 경우, 하나의 대표 항목으로 통합된 결과를 확인할 수 있다. 단순 중복과 수정본(리비전)은 구분된다.

**Why this priority**: 중복 리포트를 걸러내야 사용자가 실질적으로 유의미한 결과만 볼 수 있다. 수집(P1) 이후 동작하며, 요약(P2)과는 독립적으로 테스트 가능하다.

**Independent Test**: 동일 리포트가 HTML과 PDF 형태로 2개 사이트에서 수집된 케이스에서, 결과가 1건으로 통합되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 동일 리포트가 HTML과 PDF 형태로 2개 사이트에서 수집되었을 때, **When** 중복 제거가 완료되면, **Then** 하나의 canonical 항목으로 통합되어 표시된다.
2. **Given** 동일 제목의 리포트가 수정본으로 재발행되었을 때, **When** 중복 판별을 수행하면, **Then** 수정본은 별도로 표시되되 최신 버전이 primary로 취급된다.
3. **Given** 제목과 증권사는 같지만 다른 종목에 대한 리포트일 때, **When** 중복 판별을 수행하면, **Then** 별개 리포트로 유지된다.

---

### User Story 4 - 분류별 탐색 (Priority: P4)

사용자는 수집된 리포트를 증권사별, 종목별, 산업별, 애널리스트별로 그룹화하여 탐색할 수 있다. 동일 종목에 대한 여러 증권사 리포트를 한 번에 비교할 수 있다.

**Why this priority**: 리포트를 원하는 기준으로 필터링/그룹화하는 것은 탐색 효율성을 높이는 부가 가치이다. 수집(P1)과 메타데이터 추출이 선행되어야 한다.

**Independent Test**: 수집된 리포트 10건에 대해 종목별, 증권사별 그룹화를 실행하고 결과가 올바르게 분류되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 메타데이터가 추출된 리포트 목록이 있을 때, **When** 증권사별 그룹화를 요청하면, **Then** 동일 증권사의 리포트가 하나의 그룹으로 묶여 표시된다.
2. **Given** 동일 종목에 대해 3개 증권사의 리포트가 있을 때, **When** 종목별 그룹화를 요청하면, **Then** 해당 종목 아래 3개 리포트가 함께 표시된다.
3. **Given** 산업/섹터 정보가 있는 리포트들에 대해, **When** 산업별 그룹화를 요청하면, **Then** 동일 산업의 리포트가 그룹으로 묶인다.

---

### User Story 5 - 최종 결과물 생성 (Priority: P5)

사용자는 하루의 전체 리포트 현황을 하나의 통합 결과물로 받아볼 수 있다. 결과물은 사람이 읽기 좋은 요약 형태와 프로그래밍 방식으로 소비 가능한 구조화 데이터 두 가지 형태로 제공된다.

**Why this priority**: 모든 처리(수집, 검증, 중복 제거, 요약, 분류)의 최종 산출물이므로, 앞선 기능이 모두 갖춰진 후 의미 있는 결과를 낸다.

**Independent Test**: 처리 완료된 리포트 세트에 대해 두 가지 형태(사람 읽기용, 구조화 데이터)의 결과물이 모두 생성되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 모든 파이프라인 처리가 완료되었을 때, **When** 결과 생성을 실행하면, **Then** 사람이 읽기 좋은 요약 리스트(Markdown/HTML)가 생성된다.
2. **Given** 모든 파이프라인 처리가 완료되었을 때, **When** 결과 생성을 실행하면, **Then** 구조화 데이터(JSON) 형태의 결과물이 함께 생성된다.
3. **Given** 결과물이 생성되었을 때, **When** 사용자가 확인하면, **Then** 투자자가 5분 이내에 하루 리포트 현황을 파악할 수 있는 수준의 정보가 포함되어 있다.

---

### Edge Cases

- 특정 사이트가 일시적으로 접근 불가할 때: 해당 사이트는 건너뛰고 나머지 사이트에서 수집을 계속 진행하며, 실패 사이트를 별도로 기록한다.
- 특정 날짜에 발행된 리포트가 하나도 없을 때: 빈 결과를 반환하되, "해당 날짜에 수집된 리포트가 없습니다" 메시지를 표시한다.
- 리포트의 발행일이 날짜 경계에 걸칠 때 (예: 23:50 KST 발행): Asia/Seoul 타임존 기준으로 날짜를 판단한다.
- 동일 리포트가 3개 이상의 사이트에서 수집될 때: 모든 중복을 하나의 canonical 항목으로 통합하고, 각 출처 URL을 모두 기록한다.
- 리포트 본문이 유료 콘텐츠로 차단되어 있을 때: 해당 리포트는 메타데이터만으로 최종 결과에 포함하고, 요약 필드는 "본문 접근 불가"로 표시한다. 우회 시도하지 않는다.
- 사이트 구조가 변경되어 파서가 동작하지 않을 때: 파서 오류를 기록하고, 해당 사이트 결과를 건너뛴다. 전체 파이프라인은 계속 진행한다.
- 수집 도중 rate limit에 도달할 때: 해당 사이트의 수집을 즉시 중단하고, 이미 수집된 데이터로 계속 처리한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 명시적인 target_date 파라미터를 받아 해당 날짜 기준으로 리포트를 수집해야 한다.
- **FR-002**: 시스템은 사전에 승인된 사이트 목록(allowlist)에 등록된 출처에서만 리포트를 수집해야 한다.
- **FR-003**: 시스템은 HTML 페이지, PDF 파일 등 다양한 형태의 리포트를 수집할 수 있어야 한다.
- **FR-004**: 시스템은 수집된 각 리포트의 발행일을 본문, 메타데이터 등에서 추출하고, target_date와 일치하는 경우에만 최종 결과에 포함해야 한다.
- **FR-005**: 날짜를 추출할 수 없거나 불명확한 리포트는 최종 결과에서 제외하고, UNVERIFIED 상태로 별도 표시해야 한다.
- **FR-006**: 시스템은 각 기업분석(종목) 리포트에서 다음 메타데이터를 구조화하여 추출해야 한다: 제목, 종목명/코드, 증권사명, 애널리스트명, 발행일, 원본 URL.
- **FR-007**: 필수 메타데이터(제목, 증권사, 애널리스트, 발행일, URL) 중 하나라도 누락된 리포트는 최종 결과에 포함하지 않아야 한다.
- **FR-008**: 시스템은 동일 리포트(동일 콘텐츠의 PDF, HTML, 재게시 등)를 하나의 canonical 레코드로 통합해야 한다.
- **FR-009**: 시스템은 단순 중복과 수정본(리비전)을 구분하고, 수정본은 최신 버전을 primary로 취급해야 한다.
- **FR-010**: 시스템은 각 리포트에 대해 주요 내용, 투자 의견, 핵심 포인트를 포함한 요약을 생성해야 한다.
- **FR-011**: 요약 생성 시 원문에 없는 수치(목표주가, 실적 등)를 임의로 생성하지 않아야 한다. 정보가 없으면 "정보 없음"으로 표시해야 한다.
- **FR-012**: 요약 결과에서 원문 추출 정보(extracted)와 생성 요약(generated)을 구분하여 표시해야 한다.
- **FR-013**: 시스템은 수집된 리포트를 종목별, 산업별, 증권사별, 애널리스트별로 그룹화할 수 있어야 한다.
- **FR-014**: 시스템은 최종 결과를 사람이 읽기 좋은 요약 형태(Markdown/HTML)와 구조화 데이터(JSON) 두 가지로 제공해야 한다.
- **FR-015**: 시스템은 각 사이트의 robots.txt, 서비스 약관, rate limit을 준수해야 한다.
- **FR-016**: 시스템은 유료 콘텐츠 또는 접근 차단된 콘텐츠를 우회하지 않아야 한다.
- **FR-017**: 시스템은 파이프라인의 각 단계(수집, 파싱, 검증, 중복제거, 요약, 출력)에 대한 구조화된 로그를 기록해야 한다.
- **FR-018**: 시스템은 개별 사이트 또는 agent의 실패가 전체 파이프라인의 실패로 이어지지 않도록 격리하여 처리해야 한다.
- **FR-019**: 시스템은 중간 단계에서 중단된 경우 해당 단계부터 재시작 가능한 구조를 가져야 한다.
- **FR-020**: 모든 날짜/시간 처리는 Asia/Seoul(KST, UTC+9) 타임존 기준이어야 한다.
- **FR-021**: 동일한 target_date와 동일한 입력 데이터에 대해 동일한 결과를 산출해야 한다 (결정적 처리).
- **FR-022**: 증권사명, 종목명/코드 등은 표준화된 형태로 정규화해야 한다.
- **FR-023**: 본문에 접근할 수 없는 리포트(유료 차단, PDF 다운로드 불가 등)는 메타데이터만으로 최종 결과에 포함하되, 요약 필드는 "본문 접근 불가"로 표시해야 한다.

### Key Entities

- **Report**: 개별 주식 리포트. 제목, 종목, 증권사, 애널리스트, 발행일, 원본 URL, 콘텐츠 형태(HTML/PDF), 검증 상태를 포함한다.
- **Site**: 수집 대상 사이트. 사이트명, URL, 파서 유형, rate limit, 인증 방식, 활성 상태를 포함한다. allowlist에 등록된 사이트만 수집 대상이 된다.
- **Summary**: 리포트 요약. 원문 추출 정보(extracted)와 생성 요약(generated)으로 구분된다. 주요 내용, 투자 의견, 목표주가, 핵심 포인트를 포함한다.
- **DailyResult**: 특정 날짜의 전체 처리 결과. 대상 날짜, 수집된 리포트 수, 검증 통과 수, 중복 제거 후 최종 수, 사람 읽기용 결과물과 구조화 데이터를 포함한다.
- **PipelineLog**: 파이프라인 각 단계의 처리 기록. timestamp, 단계명, 입력 항목 ID, 출력 항목 ID, 처리 상태, 에러 정보를 포함한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 하루치 리포트 현황을 결과물을 통해 5분 이내에 파악할 수 있어야 한다.
- **SC-002**: 수집된 리포트의 날짜 검증 정확도는 95% 이상이어야 한다 (잘못된 날짜의 리포트가 포함되는 비율 5% 미만).

## Assumptions

- 사용자는 시스템을 커맨드라인 또는 스크립트를 통해 실행하며, 웹 UI는 v1 범위에 포함하지 않는다.
- v1 수집 대상 사이트는 3곳: (1) 네이버 리서치 (https://finance.naver.com/research/company_list.naver), (2) 한경컨센서스, (3) 증권사 직접 리서치센터 1곳. 이후 점진적으로 확장한다.
- 리포트는 주로 한국어로 작성되어 있으며, 요약 결과도 한국어로 출력한다.
- 시스템은 단일 머신에서 실행되며, 분산 처리는 v1 범위에 포함하지 않는다.
- LLM 기반 요약 생성은 외부 API를 호출하며, 해당 API에 대한 접근 권한이 확보되어 있다.
- 수집 대상 사이트의 구조는 비교적 안정적이나, 변경 시 파서 업데이트가 필요할 수 있다.
- 리포트의 발행일 정보는 대부분의 경우 사이트 페이지 또는 파일 메타데이터에서 추출 가능하다.
- v1에서 수집 대상은 기업분석(종목) 리포트만 해당한다. 산업/섹터 리포트, 매크로/전략, 채권 리포트는 v1 범위에 포함하지 않는다.
- 결과 데이터의 장기 보관 및 이력 관리는 v1 범위에 포함하지 않는다.
